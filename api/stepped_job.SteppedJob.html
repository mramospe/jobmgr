SteppedJob¶

class stepped_job.SteppedJob(path=None, registry=None)[source]¶

Bases: stepped_job.jobs.JobBase

Instance to handle different steps, linked together. This object creates a new directory under “path” with a job ID. This job ID is expected to be a number, and the next to the greatest in the directory will be associated to the job (0 if none is found).

Parameters:
  • path (str) – path to the desired directory.
  • registry (JobRegistry or None) – instance to register the object. If “None”, the object will be registered in the main JobManager instance.
Variables:
  • steps – Steps managed by this job.
  • jid – Job ID, determined by the subdirectories in the output path.

add_step(name, executable, opts, data_regex, data_builder=None)[source]¶

Add a new step to the job, concatenating it with the previous one.

Parameters:
  • name (str) – name of the step.
  • executable (str) – application/version to run.
  • odir (str) – where to create the output directory.
  • data_regex (str) – regex representing the output data to send to the next step.
  • data_builder (function) – function to define the way how the data is passed to the executable. It must take a list of strings (paths to the data files), and return a merged string. The default behaviour is to return list to be executed as: <executable> file1 file2 … but one can also define the string so the argument is explicitely specified, like: <executable> –files file1 file2 …
Raises:

RuntimeError – if the name used for this step is already being used in another step.
